## D-001 — Нет собственного UI, интерфейс в Яндекс AI Studio

**Дата:** 2026-05-27
**Контекст:** Старый проект (agentKP) имел собственный frontend на Netlify. Это создавало сложности с деплоем, лимитами и поддержкой UI-логики.

**Принятое решение:** Backend предоставляет только HTTP-инструменты. Весь UI и диалог с пользователем — в Яндекс AI Studio через MCP-протокол.

**Последствия:**
- ✅ Можно: добавлять новые /tools/* эндпоинты
- ❌ Нельзя: добавлять frontend папку или HTML-страницы
- ❌ Нельзя: хранить состояние диалога на backend — это задача Студии

**Статус:** Активно

## D-002 — Новый репозиторий, не ветка agentKP

**Дата:** 2026-05-27
**Контекст:** Старый проект agentKP накопил технический долг: монолитный /process, смешанная логика сессий и UI, непоследовательная архитектура после спринтов 1-2.

**Статус:** Активно

## D-003 — ЕСКЛП в DuckDB in-memory, загружается при старте

**Дата:** 2026-05-27
**Контекст:** ЕСКЛП содержит ~500К+ записей КЛП. Нужен быстрый поиск по торговому названию.

**Принятое решение:** При старте сервиса загружаем только `tn_smnn_*.xlsx` (торговые наименования → МНН) в DuckDB in-memory. Это ~28К строк, загружается за секунды.

**Последствия:**
- ✅ Можно: перезагружать ЕСКЛП рестартом сервиса
- ❌ Нельзя: хранить ЕСКЛП в PostgreSQL или отдельном файле на диске
- ❌ Нельзя: коммитить файлы ЕСКЛП в git

**Статус:** Активно

## D-004 — Список 295 ПК в памяти, передаётся LLM целиком

**Дата:** 2026-05-27
**Контекст:** 295 подкатегорий — статичный список, не меняется. Нужно передавать LLM для выбора ПК.

**Принятое решение:** `pk_list.json` загружается в память при старте. При каждом вызове `/tools/match_pk` весь список передаётся в промпт LLM. 295 ПК ≈ 3-5К токенов — допустимо.

**Последствия:**
- ✅ Можно: обновить список перезапуском сервиса
- ❌ Нельзя: использовать векторный поиск или embedding — избыточно для 295 элементов

**Статус:** Активно

## D-005 — Три уровня поиска ПК: fuzzy МНН → LLM с 295 ПК → LLM без МНН

**Дата:** 2026-05-27
**Контекст:** МНН в базе нестандартное (может быть шире: «статины» вместо «аторвастатин», или уже: «аторвастатин 10мг»). Для БАДов и медизделий МНН отсутствует.

**Принятое решение:**
1. Fuzzy-match МНН из ЕСКЛП по pk_list (rapidfuzz >= 80) → если нашли с confidence >= 0.90: auto_matched
2. Если confidence < 0.90 или МНН нестандартное → LLM с полным списком 295 ПК
3. Если МНН пустое (БАД, медизделие, косметика) → сразу LLM по торговому названию и типу товара

**Статус:** Активно

## D-007 — ЕСКЛП enrich: АТХ и ФТГ из esklp_smnn для match_pk

**Дата:** 2026-05-28
**Контекст:** Поиск ЕСКЛП только по `tn_smnn` возвращал МНН, форму и дозировку. Этого недостаточно для выбора ПК, потому что одно действующее вещество может относиться к разным областям применения. Пример: гиалуроновая кислота встречается и в глазных каплях (АТХ `S01`), и в протезах суставов (АТХ `M09`). Без АТХ/ФТГ LLM может отправить суставной препарат вроде Армавискона в офтальмологию.

**Принятое решение:** Дополнительно загружать `esklp_smnn_*.xlsx` в DuckDB таблицу `esklp_smnn` и связывать её с `tn_smnn` по `smnn_code`. Из `esklp_smnn` брать `atx_code`, `atx_name`, `ftg_name` и возвращать эти поля из `/tools/search_esklp`. При вызове `/tools/match_pk` эти поля входят в payload для LLM вместе с МНН, формой и дозировкой.

**Последствия:**
- ✅ Можно: использовать АТХ и ФТГ как классификационные признаки при выборе ПК.
- ✅ Нужно: читать рабочие листы `tn_smnn_*` и `esklp_smnn_*`, потому что первый лист в файлах ЕСКЛП — `Описание`.
- ❌ Нельзя: передавать LLM только МНН без классификационного контекста, если из ЕСКЛП доступны АТХ/ФТГ.

**Статус:** Активно
